# Netplay Alias Generation Pipeline

This document explains how RetroVault automatically groups ROM variants for
netplay compatibility, so players on different regional or revision releases
can join the same room.

## Overview

When two players launch the same game in different versions (e.g., Pokémon
FireRed USA vs. Pokémon FireRed Japan), they need to land in the same netplay
lobby.  The alias pipeline assigns every known ROM variant a canonical
**room key** so all compatible versions hash into the same room.

```
"Pokemon FireRed (USA)"  ──┐
"Pokemon FireRed (Europe)" ─┼──► pokemon_gen3_kanto
"Pokemon LeafGreen (JP)"   ─┘
```

## Components

### 1 — Runtime resolver (`src/multiplayer.ts`)

The main TypeScript module provides the public API:

| Function | Description |
|---|---|
| `normalizeRomTitle(title)` | Strip region/revision tags; return lowercase space-separated title |
| `canonicalizeGameId(title)` | Same as above but returns underscore-separated filesystem-safe key |
| `stripRegionSuffix(title)` | Remove `(USA)`, `(Europe)`, `(Japan)`, … tags |
| `stripRevisionSuffix(title)` | Remove `(Rev 1)`, `(v1.1)`, … tags |
| `resolveNetplayRoomKey(gameId, systemId?)` | Full resolution: normalize → alias lookup → fallback to canonical key |
| `validateAliasTable()` | Integrity check for duplicate/overlapping alias rules |

#### Example

```typescript
import { resolveNetplayRoomKey, normalizeRomTitle } from './multiplayer';

normalizeRomTitle('Pokemon - FireRed Version (USA) (Rev 1)');
// → "pokemon firered version"

resolveNetplayRoomKey('Pokemon FireRed (USA)', 'gba');
// → "pokemon_gen3_kanto"

resolveNetplayRoomKey('Pokemon FireRed (Europe)', 'gba');
// → "pokemon_gen3_kanto"  ✓ same room
```

### 2 — Compatibility alias table (`src/compatibility_aliases.json`)

The runtime alias table maps room keys to regex patterns that match
canonicalized game IDs.  It is imported directly by `multiplayer.ts` at
module load and compiled into `RegExp` objects for sub-millisecond lookups.

```json
{
  "pokemon_gen3_kanto": [
    "^(?:pokemon|pok[eé]mon|pocket[_\\s]*monsters)_(?:fire_?red|leaf_?green)(?:_version)?$"
  ]
}
```

**Format:** `{ "<room-key>": ["<regex-pattern>", …] }`

The system bucket (gbc / gba / nds) is inferred from the room key name:

| Room key contains | System bucket |
|---|---|
| `gen1` or `gen2` | `gbc` (and `gb`) |
| `gen3` | `gba` |
| anything else | `nds` |

### 3 — Generated alias map (`src/generated_netplay_aliases.json`)

A human-readable JSON file listing the known normalized title strings for each
room key.  Generated by the build pipeline and committed as a reference; also
used by the debug CLI.

```json
{
  "pokemon_gen3_kanto": [
    "pokemon fire red",
    "pokemon firered",
    "pokemon leaf green",
    "pokemon leafgreen"
  ]
}
```

**Format:** `{ "<room-key>": ["<normalized-title>", …] }`

Regenerate after adding new No-Intro DAT data:

```bash
node tools/generate-netplay-aliases.js
```

## Tools

### `tools/parse-nointro-dat.js`

Parses No-Intro ROM database DAT files (Logiqx XML or ClrMamePro text format)
into structured JavaScript objects.

```bash
# Print a summary of entries in a DAT file
node tools/parse-nointro-dat.js "./No-Intro GBA.dat"

# Dump full parsed data as JSON
node tools/parse-nointro-dat.js "./No-Intro GBA.dat" --json
```

Each returned entry has:

```typescript
{
  name:     string,   // Full ROM title e.g. "Pokemon - Fire Red Version (USA)"
  romName:  string,   // ROM filename
  size:     number,   // File size in bytes
  crc:      string,   // CRC32 checksum
  sha1:     string,   // SHA-1 checksum
  region:   string,   // Extracted region tag e.g. "usa"
  revision: string,   // Extracted revision tag e.g. "Rev 1"
  serial:   string,   // Game serial if present
}
```

### `tools/generate-netplay-aliases.js`

Generates `src/generated_netplay_aliases.json` from No-Intro DAT files and/or
the built-in compatibility_aliases.json seed table.

```bash
# Generate from built-in table only (no DAT files needed)
node tools/generate-netplay-aliases.js

# Enrich with a real No-Intro GBA DAT file
node tools/generate-netplay-aliases.js "./No-Intro GBA.dat"

# Custom output path
node tools/generate-netplay-aliases.js --out ./my-aliases.json
```

### `tools/netplay-alias-debug.js`

Debug a single game title through the full alias pipeline.

```bash
node tools/netplay-alias-debug.js "Pokemon FireRed (USA)"
node tools/netplay-alias-debug.js "Pokemon FireRed (USA)" --system gba
```

**Output:**

```
Input title:      Pokemon FireRed (USA)
Normalized title: pokemon firered
Canonical key:    pokemon_firered
Alias group:      pokemon_gen3_kanto
Room key:         pokemon_gen3_kanto
System:           gba
Room hash:        0x…
```

## How No-Intro data is used

No-Intro maintains authoritative ROM databases for many systems.  Typical DAT
entry:

```xml
<game name="Pokemon - Fire Red Version (USA)">
  <rom name="Pokemon - Fire Red Version (USA).gba" size="…" crc="…" sha1="…"/>
</game>
```

The pipeline:

1. **Parse** the DAT file with `parse-nointro-dat.js`.
2. **Normalize** each game name with `normalizeRomTitle()` — strips region/
   revision tags, diacritics, punctuation → `"pokemon fire red version"`.
3. **Match** the canonical key against `compatibility_aliases.json` regex
   patterns → room key `"pokemon_gen3_kanto"`.
4. **Group** all matched normalized titles under the same room key.
5. **Write** `generated_netplay_aliases.json` with the grouped titles.

The regex patterns in `compatibility_aliases.json` are the source of truth for
what counts as a compatible group; No-Intro data enriches the output with real
variant names.

## Adding custom compatibility groups

### Step 1 — Add a regex rule to `compatibility_aliases.json`

```json
{
  "my_game_group": [
    "^my_game_(?:red|blue|green)(?:_version)?$"
  ]
}
```

The room key naming convention:
- lowercase, underscore-separated
- for Pokémon: `pokemon_gen<N>_<region>` pattern
- for others: `<title>_<variant>` pattern

The system bucket is inferred automatically from the room key name (see §2
above).  For explicit control, add a `"_sys_<system>_"` segment to the key, or
patch the inference logic in `multiplayer.ts`.

### Step 2 — Add a display name (optional)

In `multiplayer.ts`:

```typescript
export const ROOM_KEY_DISPLAY_NAMES: Record<string, string> = {
  // … existing entries …
  my_game_group: "My Game Trading Room",
};
```

### Step 3 — Regenerate the alias map

```bash
node tools/generate-netplay-aliases.js
```

### Step 4 — Add tests

```typescript
it('groups My Game variants under my_game_group', () => {
  expect(resolveNetplayRoomKey('My Game Red', 'gba')).toBe('my_game_group');
  expect(resolveNetplayRoomKey('My Game Blue', 'gba')).toBe('my_game_group');
});
```

## Supported systems

| System | Link capability | Notes |
|---|---|---|
| `gb`  | ✓ | Original Game Boy; shares GBC alias rules |
| `gbc` | ✓ | Game Boy Color |
| `gba` | ✓ | Game Boy Advance |
| `nds` | ✓ | Nintendo DS |
| `n64` | ✓ | Nintendo 64 |
| `psp` | ✓ | PlayStation Portable |
| `nes` | ✗ | No link-cable multiplayer |
| `snes` | ✗ | No link-cable multiplayer |

## Performance

Alias resolution runs in **< 1 ms** per lookup:

- The alias table is compiled once at module load into pre-built `RegExp` objects.
- Confidence scoring uses a single regex test on the canonical key.
- There are no async operations in the hot path.
- `canonicalizeGameId` caches nothing but is a pure string transformation with
  O(n) complexity in title length.
